# Phase 2 · 结构化确认

> 目标：将Phase 1收集的全部信息整理为Case State A+B层，呈现给用户逐项确认
> 这是进入自动分析前的最后检查点——确认后Phase 3全自动执行
> 一轮完成

## 执行逻辑

将Phase 1填充的Case State A+B层整理为标准化的确认表，呈现给用户。

## 确认表格式

```
═══════════════════════════════════════
  信息确认表 · 请逐项确认后我将开始分析
═══════════════════════════════════════

【交易主体】
┌─────────┬──────┬──────┬──────┬──────┬──────┐
```

## 确认规则

- 用户确认（"确认"/"没问题"/"对的"）→ A+B层锁定 → 进入Phase 3
- 用户修正某项 → 更新Case State → 重新生成确认表 → 再次等待确认
- 用户补充信息 → 更新Case State → 如果补充改变了事件识别或方案方向 → 重新呈现确认表
- 用户追问/有疑问 → 先回答 → 再呈现确认表等待确认
- 用户不确认 → 不进入Phase 3，主动提示

## 确认表的质量标准

- **完整性**：Phase 1收集到的每一项信息都必须出现在确认表中，不遗漏
- **追溯可见**：成本基础的计算链路要展示清楚（如"张女士初始480万→转出20%给父亲(160万)→剩余320万→增资后仍为320万"）
- **缺失标注**：门禁已通过但仍有缺失的非门禁信息，标注"未获取，将按[默认假设]处理"
- **初步判断有理由**：不只写"方案A值得评估"，要写清为什么（"未分配利润2000万，法人间分红免税通道可降低转让税基"）
